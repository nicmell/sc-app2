# `@sc-app/ui-components`

The sc-app design system: **design tokens** (authored as JSON, generated by
[Style Dictionary](https://styledictionary.com) into CSS custom properties **and**
a typed TS module) plus a library of framework-agnostic **`-base` components** —
Lit web components that each style themselves in their own **shadow root** (Lit
`css`), with React wrappers.

Two consumers:

1. **The React host** (`src/`) — imports the global foundation CSS once in
   `src/main.tsx` (tokens + reset + base), registers the web components, uses the
   React wrappers, and can import token values from `@sc-app/ui-components/tokens`.
2. **Lit widgets** (`src/sc-elements/*`) — render the `-base` tags directly.

(Each component is self-contained — its styles live in its shadow — so there's
no global component-class layer for raw HTML to consume; use the elements.)

## Layout

```
tokens/                    design-token source (JSON): space, radius, typography,
                           elevation, motion, layout, color
build-tokens.mjs           Style Dictionary build (`yarn tokens`)
src/
  foundations/             the GLOBAL stylesheet — tokens + reset + base only
    index.css              entry; @imports the generated tokens + reset + base
    generated/tokens.generated.{css,ts}   GENERATED — :root vars + typed TS module
    base/{elements,typography}.css   bare button/input/select/textarea/label/headings
    reset.css
  components/              the -base Lit web components (one file each)
    index.ts               element barrel + registerUiComponents()
    <name>.ts + <name>.styles.ts   element + its shadow styles (Lit `css`)
    internal/{sc-widget-base,sc-dialog-base,reset.styles,widget-base.styles,…}
    react.ts               all @lit/react wrappers (one-liners) in a single file
```

### Entry points (package `exports`)

| import | what |
|---|---|
| `@sc-app/ui-components` | the global foundation CSS (tokens + reset + base) — load once at app boot |
| `@sc-app/ui-components/tokens` | the typed TS token module (`import { color, space } from …`) |
| `@sc-app/ui-components/tokens.css` | just the generated `:root` custom properties |
| `@sc-app/ui-components/lit` | the web components + `registerUiComponents()` |
| `@sc-app/ui-components/react` | the React wrappers (importing the barrel registers the elements) |
| `@sc-app/ui-components/reset` | the reset layer on its own |

## Using the components

```ts
// 1. styling (once, at app boot) — the global foundation CSS: design tokens
//    (:root custom properties, which INHERIT into every component's shadow root)
//    + reset + base element styles. No per-component CSS — each component ships
//    its own styles in its shadow.
import "@sc-app/ui-components";

// 2a. as web components (Lit)
import { registerUiComponents } from "@sc-app/ui-components/lit";
registerUiComponents();            // idempotent; defines every <sc-*-base> tag
// → <sc-button-base label="Run" variant="danger"></sc-button-base>

// 2b. as React components
import { ScButton, ScSelect } from "@sc-app/ui-components/react";
// → <ScButton label="Run" variant="danger" onClick={…} />

// 3. token values in JS (same source as the CSS vars)
import { color, space } from "@sc-app/ui-components/tokens";
ctx.fillStyle = color["surface-2"]; // e.g. a canvas colour matching the palette

// 4. icons: load the Phosphor fill font once (peer dep) — registers the
//    @font-face so the glyphs resolve inside <sc-icon-base>'s shadow.
import "@phosphor-icons/web/fill";
```

All components are **shadow DOM** and self-styled; the tokens reach them by
inheritance from `:root`. **Events are native**: each form widget wraps a native
`<input>` and re-emits its `input`/`change` as a **composed** event off the host,
so consumers read `e.target.value` / `e.target.checked` (the target is the
element itself). The React wrappers expose `onChange` (+ `onInput` for live
controls). Containers (`sc-select-base`, `sc-radio-group-base`) coordinate their
declarative children via Lit context (composed, so it crosses shadow boundaries)
and fire a plain `change` from the host. `sc-button-base` uses the native `click`
(React `onClick`); `sc-toast-base` adds `dismiss`/`onDismiss`.

**Form participation:** every value-bearing widget is a form-associated custom
element (`ElementInternals`), so it submits in a `<form>` under its `name` just
like a native control.

### Variant vocabularies (intentionally different)

- **Widgets** (checkbox/switch/knob/slider/radio/select via `ScWidgetBase`) —
  `variant` is a colour **accent**: `primary` (default) `neutral` `ok` `warn` `danger`.
- **`sc-button-base`** — `variant` is an **appearance**: `primary` (default)
  `secondary` `ghost` `danger`.
- **`sc-badge-base`** — `ok` (default) `warn` `error`.
- **`sc-chip-base`** — `neutral` (default) `ok` `warn` `error` `info`.
- **`sc-toast-base`** — `default` `success` `warn` `error` `info`.
- **`sc-text-base`** — `tone`: `default` `dim` `mute` `faint` `primary` `ok` `warn` `error` `info`.
- **`sc-alert-base`** — `variant`: `info` (default) `success` `warn` `error`.

`size` is `sm | md | lg` everywhere it appears (md default). `gap` (stack/
cluster) is a monotonic `xs | sm | md | lg` mapping 1:1 to `--space-{xs,sm,md,lg}`
(8 / 12 / 16 / 20px), default `xs`.

## Architecture

Every `-base` component is a **shadow-DOM** Lit element that owns its styles via
Lit `css` — a co-located `<name>.styles.ts` exporting a `CSSResult`, composed
into `static styles`. The design **tokens** live once on `:root` (the generated
`tokens.generated.css`, loaded by the app) and **inherit through the shadow
boundary**, so a component just references `var(--space-md)` — no per-component
token adoption. Shared style fragments are CSSResults too: `internal/reset.styles.ts`
(the box-sizing + form-font reset a shadow root needs, since the document reset
doesn't cross the boundary) and `internal/widget-base.styles.ts` (the form-widget
family's `.sr-only` / disabled affordance / variant→`--_accent`).

Rendering shapes:

1. **Content wrappers** (text, alert, panel, empty, stack, cluster) render a
   `<slot>` and style `:host` / `:host([attr])` / `::slotted(…)` off reflected
   props (`:host([variant="warn"])`, `:host([gap="md"])`).

2. **Form widgets** (checkbox, switch, knob, slider, radio, input, inputnumber,
   textarea) wrap a native control (`.sr-only` under an SVG/CSS overlay for the
   toggles/ranges). The native input owns value + keyboard; the component
   **re-emits a composed `input`/`change`** off the host (native input events
   don't cross the shadow boundary) and mirrors the value through
   **`ElementInternals`** (`static formAssociated`) for `<form>` participation.
   `ScWidgetBase` (`internal/`) holds `size`/`variant`/`disabled`, `blockClasses()`,
   and the `reemit()` helper; variant/size resolve to `.sc-x--*` classes via
   `classnames` (not `[data-variant]`).

3. **Context containers** (`sc-select-base`, `sc-radio-group-base`) use
   `@lit/context` — events are composed, so the provider→consumer wiring crosses
   shadow boundaries — to coordinate their declarative children and fire a single
   `change` from the host. Providers register before consumers so static markup
   upgrades with the provider already listening.

4. **Native-element wrappers**: `sc-disclosure-base` wraps `<details>`;
   `sc-modal-base`/`sc-drawer-base` wrap `<dialog>` (shared lifecycle in
   `internal/sc-dialog-base.ts`); `sc-icon-base` adopts the Phosphor glyph CSS
   into its own shadow (an icon font's class→glyph rules don't cross the boundary;
   the `@font-face` is registered globally by the host's `@phosphor-icons/web/fill`).

### Overlays (top layer)

Dropdowns, modals, and toasts must escape clipping and stacking: an
`overflow`/`transform` ancestor (e.g. a `react-grid-layout` dashboard cell)
traps an absolutely-positioned child, and `z-index` battles are fragile. The
only correct escape is the browser **top layer**, reached two ways:

- **`PopoverController`** (`internal/popover-controller.ts`) — a Lit
  `ReactiveController` that turns a panel into a `popover="auto"` element
  (top layer + native light-dismiss: outside-click + Esc) and positions it
  against an anchor with [`@floating-ui/dom`](https://floating-ui.com)
  (`offset`/`flip`/`shift`, `strategy: "fixed"`, re-positioned via `autoUpdate`).
  Used by **`sc-popover-base`** (a generic anchored panel; anchors to its
  `anchor` property or previous sibling) and directly by **`sc-select-base`**
  (the combobox carries `popovertarget`, so the browser owns the toggle +
  light-dismiss; the controller only positions). Guarded — degrades to
  in-flow CSS positioning where the Popover API is absent.

- **Native `<dialog>`** for **`sc-modal-base`** and **`sc-drawer-base`** —
  `showModal()` gives the top layer, a `::backdrop`, a focus trap, and Esc for
  free; no anchoring, so no floating-ui. The shared open/close/dismiss lifecycle
  lives in `internal/sc-dialog-base.ts` (`dismissable` gates Esc + backdrop; a
  blocking instance swallows `cancel` and re-asserts itself if the UA
  force-closes it). The modal is centred; the **drawer** is the same dialog
  pinned to a viewport edge (`side` = right | left), sliding in/out via native
  CSS (`@starting-style` + `transition-behavior: allow-discrete`), no JS
  animation. A drawer is *edge*-anchored, not trigger-anchored — floating-ui
  would have nothing to do.

The app's **toast stack** is a `popover="manual"` element for the same reason
(never clipped). A modal `<dialog>` still renders above popovers in the top
layer, so toasts don't cover an open modal — but they're corner-placed and
modals are centred, so they don't overlap.

> **Platform note:** macOS Tauri runs in WKWebView (WebKit), where CSS anchor
> positioning isn't available — hence `@floating-ui/dom` for the math and the
> Popover API (Safari 17+) for the layer. happy-dom has no top layer or layout,
> so the open/close/positioning behaviour is verified in a real browser via the
> CDP harness; unit tests cover structure + state + events only.

## Build

```bash
yarn tokens         # Style Dictionary: tokens/*.json → generated/tokens.generated.{css,ts}
yarn typecheck      # tsc over the TS components (chained into the root typecheck)
yarn test           # vitest + happy-dom behaviour suite
```

There's no CSS bundle step: the package ships **source** — the global
`./src/foundations/index.css` (resolved by the host's Vite, which inlines the
`@import` chain) and the per-component `css` in the TS modules. The generated
token outputs are **committed**, so editing tokens means `yarn tokens` +
commit. The component `static styles` and React/Lit barrels resolve to source
through Vite aliases (decorator lowering via `esbuild.target: "es2022"`).

## Demo

`demo.html` renders the foundation + every `-base` component. The components need
the TS modules transpiled and decorators lowered, so **serve it through Vite**:

```bash
npx vite            # from packages/ui-components/, then open the printed URL
```

## Constraints

- **Tokens are the source of truth.** Author them in `tokens/*.json`; renaming a
  generated `--color-*` / `--space-*` token is a breaking change — add freely,
  rename only with a major bump.
- **Each component owns its styles** (shadow DOM + Lit `css`). There's no global
  component-class layer; the global stylesheet is only tokens + reset + base
  elements + a few app-level layout classes.
- **`-base` components are UI-only.** No OSC, store, or bind logic — the logical
  `sc-elements` in the host app wrap them to add behaviour.
